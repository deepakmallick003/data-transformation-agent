## 1. Introduction

### 1.1 Purpose and Background

This document specifies the data feed used to ingest customs declaration PDF documents from the HMRC
customs document archive hosted in Amazon S3 and prepare them for structured loading into the
Snowflake analytics platform.

### 1.2 Scope

This feed covers:

- customs declaration PDF documents stored in S3
- JSON manifest records describing the PDFs
- declaration-level index records used for routing and reconciliation

### 1.3 Audience

- Data Engineers
- Customs Analytics Engineers
- Platform Operations
- Business Analysts (Border and Trade)

---

## 2. Description

The feed processes daily customs declaration documents generated by CDS-related services. The
primary source artifacts are PDF documents stored in S3, accompanied by a JSON manifest and a CSV
index file. The target platform is Snowflake, where structured declaration headers, commodity-line
details, and ingestion audit outcomes are stored for reporting and exception management.

**Assumption:** PDF field extraction is performed by the implementation layer rather than by a
pre-existing OCR service output feed.

---

## 3. Data Feed Extract & Load Process

The extract and load process follows this pattern:

1. Retrieve the manifest object from S3 for the business date.
2. Read the declaration index file for routing and expected object counts.
3. Download or stream each PDF object from the declared S3 prefix.
4. Extract declaration data from the PDF content.
5. Validate the extracted content against business and reference rules.
6. Load accepted rows into Snowflake curated tables.
7. Write rejected rows and ingestion diagnostics to the Snowflake rejection table.

**Assumption:** S3 access is read-only for the code-generation use case and Snowflake writes occur
through a service account.

## 5. Data Feed Structure

### 5.1 Feed Content

The feed contains three source object types per business date.

#### 5.1.1 Object Listing

| Object Name | Format | Description | Example Location |
|-------------|--------|-------------|------------------|
| `manifest_YYYYMMDD.json` | JSON | Declares expected PDF keys, document counts, batch id, and submission window | `s3://hmrc-cds-doc-archive/manifests/2026/04/22/manifest_20260422.json` |
| `declaration_index_YYYYMMDD.csv` | CSV | Lightweight declaration index with MRN, declarant id, submission timestamp, and document key | `s3://hmrc-cds-doc-archive/index/2026/04/22/declaration_index_20260422.csv` |
| `*.pdf` | PDF | Customs declaration document containing declaration header and line-item details | `s3://hmrc-cds-doc-archive/raw/2026/04/22/CDSDEC-GB-20260422-0001.pdf` |

#### 5.1.2 Object Requirements

- PDFs are stored as binary objects and must be parsed by the transformation logic
- manifest files are UTF-8 JSON
- index files are pipe-delimited CSV with header row
- the manifest batch id must match the index file batch id
- only objects declared in the manifest should be processed

#### 5.1.3 Source Field Highlights

##### Manifest fields

| Field Name | Description |
|------------|-------------|
| `batch_id` | Daily submission batch identifier |
| `business_date` | Business date for the extract |
| `expected_document_count` | Count of PDFs expected for the batch |
| `document_keys` | List of S3 object keys for the PDFs |

##### Index fields

| Field Name | Description |
|------------|-------------|
| `MRN` | Movement Reference Number |
| `DOCUMENT_KEY` | S3 object key for the PDF |
| `DECLARANT_EORI` | Declarant identifier |
| `SUBMISSION_TIMESTAMP` | Submission time for the declaration |
| `DECLARATION_TYPE` | Import, export, or transit declaration category |

## 6. Source System Overview

### 6.1 Data Model

The source system is object-based rather than table-based.

```text
S3 Manifest Object
        |
        +--> Declaration Index CSV
        |
        +--> Declaration PDF Objects
```

### 6.2 Source Characteristics

| Source Component | Description |
|------------------|-------------|
| Manifest store | Tracks the expected object set for the batch |
| Index file | Provides declaration-level metadata used for routing and reconciliation |
| PDF archive | Stores declaration documents that must be parsed into structured rows |

### 6.3 Access Method

The implementation is expected to use authenticated S3 access with bucket, prefix, and region
supplied at runtime.
